# dsm_volumes

Модуль автоматизированного расчёта объёмов по цифровым моделям поверхности
(ЦМП), построенным методом фотограмметрической обработки данных
аэрофотосъёмки: склады сыпучих материалов, отвалы, участки выемки.

## Соответствие требованиям ТЗ

| Требование ТЗ | Модуль | Метод |
|---|---|---|
| Выделение границ объектов на ЦМП и ортофото | `segmentation.py` | Робастная trend-surface analysis (МНК-полином + итеративное исключение выбросов) → пороговое детектирование → watershed-разделение слипшихся объектов |
| Опорная поверхность, в т.ч. на сложном рельефе | `datum.py` | TIN-интерполяция (Delaunay/`scipy.griddata`) по кольцу точек вокруг объекта — как в Civil3D/Trimble Business Center |
| Расчёт объёмов выемки/насыпи, разновременное сравнение | `volumes.py`, `multitemporal.py` | Интегрирование (ЦМП − датум) по ячейкам; DEM of Difference между эпохами |
| Co-регистрация разновременных ЦМП | `coregistration.py` | Субпиксельная фазовая кросс-корреляция (`skimage.registration`) перед расчётом DoD |
| Фильтрация техники/конструкций | `noise_filter.py` | Площадь + форма контура (extent) + цвет по ортофото (HSV) |
| Высокое разрешение, большие площади | `tiling.py` | Тайловая параллельная обработка; безматричный (без материализации МНК-матрицы N×k) фитинг тренда — бенчмарк на реальных объёмах см. ниже |
| Отчётные документы | `reporting.py` | Ведомость объёмов (xlsx), картограмма выемки/насыпи (png), векторный слой (geojson) |
| Встраивание в конвейер как модуль | `pipeline.py`, `cli.py` | Программный API (`VolumeCalculationPipeline`) + CLI (`python -m dsm_volumes.cli`), проверено на настоящих GeoTIFF (rasterio, CRS EPSG:32642) |

## Установка

```bash
pip install numpy scipy scikit-image matplotlib pandas openpyxl pyyaml
# для чтения/записи настоящих GeoTIFF в CLI (протестировано: EPSG:32642, nodata, affine transform);
# без rasterio конвейер работает через numpy-массивы/.npy напрямую
pip install rasterio
```

## Быстрый старт

### Python API

```python
from dsm_volumes.config import PipelineConfig
from dsm_volumes.pipeline import VolumeCalculationPipeline

config = PipelineConfig(pixel_size_m=0.08)          # размер пикселя вашей ЦМП, м
pipeline = VolumeCalculationPipeline(config)

# одна эпоха съёмки
result = pipeline.run_single_epoch(dsm, ortho, out_dir="results/2026-07-01")
for v in result.volumes:
    print(v.object_id, v.kind, v.fill_volume_m3, v.cut_volume_m3)

# сравнение двух эпох (DoD)
result = pipeline.run_multitemporal(dsm_t0, dsm_t1, ortho_t1, out_dir="results/multitemporal")
```

`dsm`/`ortho` — обычные numpy-массивы (H×W float для ЦМП, H×W×3 uint8 для
ортофото); откуда они взялись (GeoTIFF через rasterio, вывод конкретного
фотограмметрического ПО и т.п.) конвейеру не важно.

### CLI (встраивание в внешний конвейер)

```bash
python -m dsm_volumes.cli \
    --dsm site_dsm.tif --ortho site_ortho.tif \
    --out-dir results/2026-07-01

python -m dsm_volumes.cli \
    --dsm site_dsm.tif --ortho site_ortho.tif --dsm-prev site_dsm_prev.tif \
    --out-dir results/multitemporal --config my_config.yaml --auto-coregister
```
CLI печатает JSON-сводку в stdout и возвращает код 0 при успехе — этого
достаточно, чтобы вызывать его одним шагом из shell-скрипта постобработки
Metashape/Pix4D/WebODM или из оркестратора (Airflow и т.п.).

## Запуск демонстрации и тестов

```bash
python3 demo_run.py     # сквозной прогон на синтетической сцене + численная валидация точности
python3 -m pytest tests/ -v
```

`demo_run.py` генерирует синтетическую сцену с ИЗВЕСТНЫМ аналитически
объёмом объектов (параболоид: V = (π/2)·H·R²) — это позволяет не просто
убедиться, что код не падает, а измерить фактическую точность каждого
алгоритма. Результаты (см. также вывод скрипта):

- восстановление датума на реалистичном уклоне+всхолмлении: ошибка **< 2%**
  (методы constant/plane/tin: +2.1% / +0.7% / +1.1%);
- на стресс-тесте (рельеф искривлён в масштабе самого объекта — уклон
  9%/6% + всхолмление с длиной волны, сопоставимой с диаметром объекта):
  constant +21.9%, plane +20.7%, **tin +7.1%** — плоскость перестаёт
  справляться, TIN остаётся точным;
- полный конвейер (детектирование → фильтр шума → объём), одна эпоха:
  **+1.6% / +0.4% / -2.7%** по трём тестовым объектам;
- мультивременное сравнение (DoD): **-0.2% / -0.5% / -0.7%** — точнее
  одноэпохного расчёта, так как не требует реконструкции рельефа
  (реальная предыдущая съёмка используется напрямую вместо восстановленного
   датума);
- имитация техники (короб сигнального цвета) корректно отсеивается фильтром
  шума и не попадает в ведомость ни в одном из режимов.

### Co-регистрация эпох (`coregistration.py`)

Если между двумя съёмками разошлась геопривязка (разные схемы GCP, дрейф
RTK/PPK) на доли/единицы пикселей — без коррекции это создаёт ложный сигнал
в DoD на любой границе перепада высоты. `run_multitemporal(auto_coregister=True)`
/ CLI `--auto-coregister` включает коррекцию субпиксельным сдвигом
(`skimage.registration.phase_cross_correlation`). Проверено на искусственно
внесённом сдвиге (1.2, -0.8) px:

| Вариант | Насыпь по площадке, м³ | Ошибка vs эталон |
|---|---|---|
| эталон (сдвига нет) | 335.4 | — |
| сдвиг есть, без коррекции | 360.8 | +7.6% |
| сдвиг есть, с коррекцией | 336.1 | +0.2% |

Честное ограничение метода: он опирается на перепады высоты самих объектов
(штабели, выемки) как источник сигнала для корреляции — на участке
абсолютно без объектов (только гладкий уклон/всхолмление) фазовая
корреляция вырождается и возвращает нулевой сдвиг независимо от реального
смещения. Для таких площадок регистрацию нужно вести по ортофото (богаче
текстурой) или по GCP/чек-пойнтам, а не по самой ЦМП — это подтверждено
отдельным тестом (см. `demo_run.py`, раздел 6, и `tests/test_coregistration.py`).

### Производительность на больших растрах (`tiling.py`)

Изолированный бенчмарк (отдельный процесс на каждый размер/метод, честная
пиковая RSS-память через `resource.getrusage`, включая накладные расходы
на генерацию тестового массива в этом же процессе) в песочнице с ~3.9 ГБ RAM:

| Размер | Пикселей | `fit_trend_surface` (плотная матрица) | `fit_trend_surface_large` (тайловая) |
|---|---|---|---|
| 2000×2000 | 4 млн | 1.74 с, 966 МБ | 1.13 с, 430 МБ |
| 4000×4000 | 16 млн | 8.47 с, 3643 МБ | 5.24 с, 951 МБ |
| 6000×6000 | 36 млн | **упал (OOM)** | 12.13 с, 1896 МБ |
| 8000×8000 | 64 млн | **упал (OOM)** | 22.46 с, 3284 МБ |

Тайловая версия не просто экономит память (не материализует МНК-матрицу
N×k) — она ещё и быстрее в 1.5-2.7 раза на всех проверенных размерах, и
единственная из двух доживает до 36-64 млн пикселей в этой же памяти.
Следующий узкий момент за пределами этого теста — сам массив ЦМП целиком
в оперативной памяти (при 100+ млн пикселей уже он один приближается к
границе ~4 ГБ); дальнейшее масштабирование требует чтения ЦМП по тем же
тайлам через `rasterio` windowed reading вместо `array[...]` в памяти —
архитектура `iter_tiles()` уже рассчитана на такую замену источника данных,
но здесь это не реализовывалось за неимением реальных данных такого объёма.

## Структура проекта

```
dsm_volumes/
    config.py         — PipelineConfig (все калибруемые параметры)
    preprocessing.py  — fill_nodata, denoise, выравнивание сеток
    segmentation.py   — детектирование объектов (trend surface + watershed)
    noise_filter.py    — отсев техники/конструкций
    datum.py            — восстановление опорной поверхности (constant/plane/tin)
    volumes.py          — интегрирование объёмов
    multitemporal.py    — DEM of Difference между эпохами
    coregistration.py    — субпиксельная co-регистрация эпох (phase cross-correlation)
    tiling.py             — тайловая обработка для больших растров
    reporting.py          — xlsx/png/geojson отчёты
    pipeline.py            — VolumeCalculationPipeline (публичный API)
    cli.py                  — точка входа для встраивания в конвейер
examples/
    synthetic_scene.py — генератор синтетической сцены для теста/валидации
tests/                  — pytest, 34 теста (сегментация, датум, объёмы,
                          фильтр шума, DoD, co-регистрация, тайлинг, сквозной прогон)
demo_run.py             — демонстрационный прогон с численной валидацией
```

## Калибруемые параметры (`PipelineConfig`)

Ключевые параметры, которые нужно откалибровать под конкретную площадку
(типичные высоты объектов, шероховатость фонового рельефа, парк техники):

- `fill_threshold_m` / `cut_threshold_m` — порог детектирования. Компромисс:
  выше — меньше ложных срабатываний на фоновой неровности, но больше
  отсечённого объёма у подошвы объекта; ниже — наоборот. Калибруется по
  остатку (ЦМП − тренд) на заведомо пустых участках площадки.
- `datum_safety_margin_m` — защитный отступ перед сбором точек грунта для
  датума (см. раздел про валидацию выше).
- `min_object_area_m2` / `machinery_*` — пороги фильтра шума, зависят от
  минимального размера легитимных объектов учёта и от парка техники,
  используемой на площадке.
